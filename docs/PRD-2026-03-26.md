# PRD - Hidraulica Gasto Web
Fecha: 26/03/2026  
Estado del producto: MVP avanzado con panel admin operativo y foco en consolidacion

## 1) Objetivo del producto
Sitio web comercial B2B para exhibir catalogo de productos de hidraulica/neumatica, captar consultas y gestionar contenido desde un panel admin de un solo usuario.

## 2) Alcance actual (caracteristicas activas)
### Sitio publico
- Home institucional con propuesta de valor, secciones de confianza y FAQ.
- Catalogo de productos con filtros, orden, busqueda y paginacion.
- Detalle de producto con galeria, datos tecnicos y CTA a WhatsApp.
- Formulario de contacto y canales de comunicacion priorizando WhatsApp.
- Mapa embebido de Google Maps en contacto/home.
- Comparador de productos.
- Notificaciones visuales (toasts) para acciones del usuario.

### Panel admin (single-admin)
- Login admin con token.
- Dashboard con resumen operativo.
- Gestion de productos:
  - Alta, edicion, eliminacion.
  - Filtros por estado/destacado/stock/categoria/marca.
  - Busqueda y orden.
  - Paginacion.
  - Acciones masivas.
  - Mostrar/ocultar precio por producto.
  - Duplicado de producto.
  - Historial de cambios por producto (versionado).
  - Edicion masiva avanzada (precio %, categoria, marca, stock, visibilidad de precio).
- Gestion de categorias y marcas.
- Gestion de consultas con triage y cambio de estado.
- Auditoria de acciones y glosario del panel.

### Plataforma tecnica
- Frontend en React + Vite.
- Backend Node/Express con endpoints admin/public.
- Feature flags para activar/desactivar funcionalidades.
- Logging cliente de errores y metricas basicas.
- Tests UI y tests funcionales del proyecto.

## 3) Avances realizados hoy
- Fix de autenticacion admin: se elimino auto-ingreso por sesion persistida; ahora pide login al entrar.
- Fix de estabilidad en panel: se evitaron crashes por estados/mocks incompletos (fallbacks seguros).
- Fix de navegacion admin: se corrigio conflicto URL/estado que hacia saltar entre tabs (dashboard/productos).
- Mejora de performance: lazy-loading por rutas con `React.lazy + Suspense`.
- Mejora de accesibilidad:
  - Link "Saltar al contenido".
  - `main` enfocable.
  - toasts con `aria-live` y roles semanticos.
  - foco visible global.
- Mejora de observabilidad:
  - metricas de navegacion (`page.view`).
  - metricas de latencia/resultado en requests API.
- Hardening UX en admin:
  - historial por producto.
  - duplicado de producto.
  - bulk avanzado.
  - robustez de estados de carga.
- Validacion tecnica:
  - `npm run test:ui` en verde.
  - `npm run build` en verde (con `SKIP_NODE_VERSION_CHECK=true` por version local de Node).

## 4) Oportunidades de mejora con valor (priorizadas)
## P0 - Alto impacto inmediato
- Migrar catalogo publico y detalle a consumo 100% API (hoy hay dependencia de mocks en partes del front).
- Endurecer auth admin:
  - manejo explicito de expiracion,
  - redirect al login ante token invalido,
  - mensaje claro de sesion vencida.
- Cerrar consistencia de textos/encoding en todas las vistas visibles.

## P1 - Valor comercial y operativo
- SEO avanzado:
  - metadata dinamica por categoria/producto,
  - schema.org completo (producto + negocio local),
  - mejoras on-page para busqueda organica.
- QA E2E de flujos criticos:
  - login admin,
  - CRUD producto,
  - triage de consultas,
  - CTAs WhatsApp.
- Mejoras de conversion:
  - CTAs persistentes por contexto,
  - microcopys orientados a contacto,
  - optimizacion de formulario de contacto.

## P2 - Escalabilidad y consolidacion
- Preparar capa de datos para migracion a DB real (sin activar hasta tener datos de cliente).
- Telemetria operativa en admin (pantalla de metricas de errores/latencia).
- Mejoras de accesibilidad WCAG (recorridos de teclado y contraste en todo el sitio).

## 5) KPI sugeridos para medir valor
- Tasa de clic a WhatsApp por pagina y por producto.
- Tiempo medio de respuesta a consultas.
- Tasa de consultas respondidas en <24h.
- Conversion de visita a contacto.
- Tiempo de operacion admin en tareas frecuentes (buscar/editar/publicar producto).

## 6) Riesgos actuales
- Version de Node local por debajo de recomendada por Vite (build funciona con bypass temporal).
- Persistencia parcial de datos mock en front hasta completar migracion API/DB.
- Cobertura E2E aun acotada para evitar regresiones de UI en cambios rapidos.

## 7) Proximo hito recomendado
Consolidar operacion admin + conversion comercial sin tocar DB:
1. Cerrar consumo API en catalogo/detalle.
2. Completar suite E2E critica.
3. Mejorar SEO y tracking de conversion.
4. Recién despues, iniciar migracion controlada a datos reales.
